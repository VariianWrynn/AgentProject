## 一、RAG 向量检索管道

**文件：** `rag_pipeline.py`

### 1.1 文档摄入流程

```
原始文档 (txt/pdf)
  → 文本清洗（去除多余空白、标准化标点）
  → ParagraphChunker 语义分块
  → SHA-256 去重（跳过已摄入内容）
  → BGE-m3 向量编码（1024维）
  → 批量写入 Milvus
```

**分块器核心逻辑（ParagraphChunker）：**
- 以段落边界（双换行 `\n\n`）为优先切分点
- 硬限制：512 tokens，50 token 重叠
- 重叠设计保证跨段落查询的语义连续性

**去重实现：**
```python
# SHA-256 指纹 → 避免重复摄入同一文本块
chunk_id = hashlib.sha256(text.encode()).hexdigest()[:16]
# 查询 Milvus 中是否已存在该 chunk_id，存在则跳过
```

### 1.2 集合 Schema

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(64) | 主键 |
| content | VARCHAR | 原始文本块 |
| embedding | FLOAT_VECTOR(1024) | BGE-m3 向量 |
| source | VARCHAR | 来源文件路径 |
| chunk_id | VARCHAR(16) | SHA-256 前16位，去重键 |
| created_at | VARCHAR | 摄入时间 |

### 1.3 检索实现

```python
# 索引：IVF_FLAT + COSINE 相似度
# 检索参数：top_k=5，score_threshold=0.45
results = collection.search(
    data=[query_embedding],
    anns_field="embedding",
    param={"metric_type": "COSINE", "params": {"nprobe": 16}},
    limit=top_k,
    output_fields=["content", "source", "chunk_id"],
)
# 过滤：score < 0.45 的结果被丢弃
return [r for r in results if r.score >= 0.45]
```

**选择 IVF_FLAT 而非 HNSW 的原因：** 语料规模较小（< 10万文档），IVF_FLAT 在精确率与构建速度上占优；HNSW 在百万级以上才有明显优势。

## 二、LangGraph 双图编排

**文件：** `langgraph_agent.py`

### 2.1 Graph 1 — ReAct 循环（/chat）

```
RouterNode → PlannerNode → ExecutorNode → ReflectorNode → CriticNode → END
```

节点间通过 `AgentState`（TypedDict）传递状态，每个节点返回 **partial update dict**，LangGraph 自动合并。

**条件边（ReflectorNode 出口）：**
```python
def _route_after_reflector(state: AgentState) -> str:
    decision   = state.get("reflection", "done")
    iteration  = state.get("iteration", 0)
    confidence = state.get("confidence", 0.0)

    if decision == "done" or confidence >= 0.7 or iteration >= MAX_ITER:
        return "critic"
    elif decision == "replan":
        return "planner"
    else:
        return "executor"    # continue
```

关键阈值：`confidence >= 0.7` 直接跳至 CriticNode，避免不必要的重复迭代；`MAX_ITER = 3` 防止无限循环。

### 2.2 Graph 2 — 多智能体管道（/research/report）

```
RouterNode → ChiefArchitect → DeepScout → DataAnalyst
           → LeadWriter → CriticMaster
           → [re_researching → DeepScout | done → Synthesizer] → END
```

**RE_RESEARCHING 条件循环：**
```python
def _route_after_critic(state: AgentState) -> str:
    phase     = state.get("phase", "done")
    iteration = state.get("iteration", 0)

    if phase == "re_researching" and iteration < 2:
        return "deep_scout"    # 补充检索
    return "synthesizer"       # 直接合成
```

触发条件：`quality_score < 0.6` AND `len(pending_queries) > 0` AND `iteration < 2`。第 3 次迭代（iteration ≥ 2）强制收敛，防止"完美主义"死循环。

### 2.3 共享单例初始化

```python
# 模块级单例，进程内复用，避免重复加载 BGE-m3 模型（~2GB）
_llm      = LLMClient()
_rag      = RAGPipeline()        # 内含 BGE-m3 SentenceTransformer
_tools    = Tools(_rag)
_text2sql = Text2SQLTool()
memgpt    = MemGPTMemory(rag=_rag)   # 复用 _rag.embed，不重新加载模型
mcp       = MCPClient()
```

## 三、六大专项 Agent 实现

**目录：** `backend/agents/`

所有 Agent 统一签名：`run(state: dict, llm: LLMClient) -> dict`，返回 partial state update。

### 3.1 ChiefArchitect — 研究规划

**文件：** `backend/agents/chief_architect.py`

职责：意图解析 → 假设生成 → 大纲规划 → 子问题拆解

**输出结构（JSON）：**
```json
{
  "hypotheses": ["假设1（含具体数字/时间节点）", ...],
  "outline": [
    {"id": "sec_1", "title": "市场概况", "description": "...", "keywords": ["kw1","kw2","kw3"]}
  ],
  "research_questions": ["子问题1", ...]
}
```

**降级机制：** 若 LLM 返回空大纲，`_default_outline()` 提供 6 个标准能源行业章节（市场概况、政策环境、竞争格局、技术趋势、数据分析、未来展望）作为兜底。

### 3.2 DeepScout — 并行深度搜索

**文件：** `backend/agents/deep_scout.py`

每个子问题触发**两路并行搜索**，所有子问题**同时并发执行**：

```python
async def _search_single(query: str) -> list[dict]:
    # Bocha 网络搜索 + RAG 知识库检索，同时发起
    web_results, rag_results = await asyncio.gather(
        _search_bocha(None, query),
        _search_rag(None, query),
    )
    return web_results + rag_results

async def _search_all(questions: list[str]) -> list[dict]:
    # 所有子问题并行，N个问题仅用1个RTT的时间
    tasks = [_search_single(q) for q in questions]
    all_results = await asyncio.gather(*tasks)
    ...
```

**去重策略：** URL 优先作为去重键；无 URL 的结果用 `md5(snippet)` 作为指纹。

**可信度评分：**
```python
def _score_credibility(item):
    base = 0.6 if item["source_type"] == "rag" else 0.5   # RAG > Web
    length_bonus = min(0.3, len(snippet) / 2000)           # 内容越长越可信
    return base + length_bonus
```

**事实提取：** 取 top-8 结果，LLM 抽取 3-8 条含具体数字/来源的结构化事实，存入 `state["facts"]`。

### 3.3 DataAnalyst — 结构化数据分析

**文件：** `backend/agents/data_analyst.py`

职责：将研究问题转化为 Text2SQL 查询，生成可视化图表。

执行链：
```
outline["keywords"] + research_questions
  → text2sql_tool.run(query)          # SQL → 结构化结果
  → matplotlib 生成图表               # 使用 SimHei 字体支持中文
  → base64 编码图表 → state["charts_data"]
  → 提取 data_points → state["data_points"]
```

### 3.4 LeadWriter — 并行章节写作

**文件：** `backend/agents/lead_writer.py`

核心：`ThreadPoolExecutor(max_workers=6)` 并行写作所有章节。

```python
def _write_one(sec: dict) -> tuple[str, str]:
    # 构造包含 facts/data_points/raw_sources 的 user_msg
    # 带 2 次重试：失败等待 2s 后重试
    for attempt in range(3):
        try:
            content = llm.chat(_SECTION_SYSTEM, user_msg, temperature=0.4)
            return sec_id, content
        except Exception:
            time.sleep(2)

max_workers = min(len(sections), 6)
with ThreadPoolExecutor(max_workers=max_workers) as executor:
    futures = {executor.submit(_write_one, sec): sec for sec in sections}
    for future in as_completed(futures):
        sec_id, content = future.result()
        draft_sections[sec_id] = content
```

全部章节完成后，单独调用 LLM 生成 200-300 字执行摘要（`draft_sections["summary"]`）。

**引用格式规范：** `[来源N]` 内联标注，要求每章至少 2 个引用，通过 system prompt 强制约束。

### 3.5 CriticMaster — 对抗式质量审核

**文件：** `backend/agents/critic_master.py`

审核 6 类问题：`hallucination | missing_source | logic_error | outdated | incomplete | bias`

**quality_score 评分标准：**
```
0.9+ → 优秀（仅低级别问题）
0.7-0.9 → 良好（若干中级别问题）
0.5-0.7 → 一般（含高级别问题）
0.0-0.5 → 不合格（严重缺陷）
```

**收敛保护逻辑：**
```python
if iteration >= 2:
    next_phase = "done"        # 强制收敛，防止无限循环
elif pending_queries and quality_score < 0.6:
    next_phase = "re_researching"
else:
    next_phase = "done"
```

阈值从 0.75 下调至 0.6，避免因过严审核频繁触发重搜循环。

**pending_queries 生成：** 仅从 `severity in ("high","medium")` 的 issue 中提取 `fix_query`，去重后最多保留 3 条。

### 3.6 Synthesizer — 报告合成

**文件：** `backend/agents/synthesizer.py`

职责：整合所有 `draft_sections` + `charts_data` + `references` + `critic_issues`（作为修订意见），输出最终 Markdown 报告。

格式模板：
```markdown
# {title}
## 执行摘要
{summary}
## {sec_1.title}
{content + 图表嵌入（base64 img标签）}
...
## 参考资料
{numbered list of references}
```

## 四、LLM 多键路由

**文件：** `llm_router.py`

### 4.1 设计动机

多个 Agent 同时调用 LLM 时，单一 API Key 会触发速率限制（rate limit）。解决方案：为每个 Agent 角色分配独立 API Key，将并发请求打散到不同配额池。

### 4.2 路由表

| Agent 角色 | 环境变量 | 并发强度 |
|-----------|---------|---------|
| router / chief_architect | LLM_KEY_1 | 低 |
| deep_scout | LLM_KEY_2 | 高（并行搜索） |
| data_analyst | LLM_KEY_3 | 中 |
| lead_writer | LLM_KEY_4 | 高（并行写作） |
| critic_master | LLM_KEY_5 | 中 |
| synthesizer | LLM_KEY_6 | 低 |

### 4.3 降级链

```python
FALLBACK_KEY = {
    "LLM_KEY_2": "LLM_KEY_1",
    "LLM_KEY_3": "LLM_KEY_1",
    "LLM_KEY_4": "LLM_KEY_1",
    "LLM_KEY_5": "LLM_KEY_1",
    "LLM_KEY_6": "LLM_KEY_1",
    "LLM_KEY_1": "OPENAI_API_KEY",   # 最终兜底
}
```

查找链：`role-key env var → KEY_1 → OPENAI_API_KEY`，三级降级保证服务永不因 key 缺失崩溃。

### 4.4 模型路由

不同角色可配置不同模型：
```python
ROLE_TO_MODEL_ENV = {
    "router":          "MODEL_ROUTER",    # 轻量分类模型
    "chief_architect": "MODEL_PLANNER",   # 规划能力强的模型
    "lead_writer":     "MODEL_WRITER",    # 写作优化模型
    "critic_master":   "MODEL_CRITIC",    # 推理严谨的模型
    ...
}
```

## 五、MemGPT 双层长期记忆

**文件：** `backend/memory/memgpt_memory.py`

### 5.1 架构概念

参考 MemGPT 论文的两层设计：

| 层次 | 存储 | 始终在上下文中 | 访问方式 |
|------|------|--------------|---------|
| Core Memory（核心记忆） | Redis | ✅ 是 | 自动注入 Planner prompt |
| Archival Memory（存档记忆） | Milvus | ❌ 否 | LLM 决策后按需检索 |

### 5.2 Core Memory — Redis 实现

```python
CORE_MEMORY_MAX = 2000   # 核心记忆总字符上限（persona + human 合计）

# Redis Key 结构
# core:{session_id}:persona  — Agent 自身描述（角色、能力）
# core:{session_id}:human    — 用户背景（职位、兴趣、公司）
```

**注入机制：** PlannerNode 在构造 system prompt 时读取并插入：
```python
core_mem = memgpt.get_core_memory(session_id)
system_with_mem = _PLANNER_SYSTEM_V2 + f"\n\n[用户背景]\n{core_mem}"
```

### 5.3 Archival Memory — Milvus 实现

```python
# 集合 Schema（archival_memory collection）
fields = [
    FieldSchema("id",         VARCHAR, max_length=64, is_primary=True),
    FieldSchema("content",    VARCHAR, max_length=2000),
    FieldSchema("session_id", VARCHAR, max_length=64),
    FieldSchema("created_at", VARCHAR, max_length=32),
    FieldSchema("embedding",  FLOAT_VECTOR, dim=1024),
]
# 索引：FLAT + COSINE（小数据量无需 IVF/HNSW）
```

**嵌入复用：** `MemGPTMemory(rag=_rag)` 传入已加载的 RAGPipeline，复用其 `embed()` 方法，避免重复加载 BGE-m3（节省 ~2GB 内存 + ~15s 启动时间）。

### 5.4 LLM 决策记忆写入

Reflector 节点执行后，由 LLM 判断是否需要更新记忆：

```python
# 支持三种操作
"core_memory_append"      → 追加用户背景/偏好到 Core Memory
"archival_memory_insert"  → 将数据结论/文档摘要存入 Archival Memory
"archival_memory_search"  → 检索历史上下文注入当前 prompt
```

触发条件：LLM 在 reflection 阶段识别到「用户提到个人背景」或「产生值得长期保存的数据结论」。

## 六、Text2SQL 三步管道

**文件：** `backend/tools/text2sql_tool.py`

### 6.1 管道步骤

```
用户问题
  → Step 1: 歧义消解（Ambiguity Detection）
  → Step 2: SQL 生成（SQL Generation）
  → Step 3: 结果解读（Summarization）
```

### 6.2 歧义消解

LLM 将中文业务术语映射为 SQL 表达式：
```python
term_dict = {
    "营收":   "SUM(amount)",
    "销售额": "SUM(amount)",
    "均价":   "AVG(unit_price)",
    "装机量": "SUM(capacity)",
    ...
}
```

非 LLM 的关键词匹配选择相关表（`sales` / `products`），降低 LLM token 消耗。

### 6.3 SQL 生成与验证

```python
# SQL 安全白名单：只允许 SELECT
if not sql.strip().upper().startswith("SELECT"):
    raise ValueError("Only SELECT statements allowed")

# 自动添加 LIMIT 50（防止返回过大结果集）
if "LIMIT" not in sql.upper():
    sql += " LIMIT 50"

# 列名存在性警告（非阻塞）
for col in extracted_columns(sql):
    if col not in schema_columns:
        logger.warning(f"Column '{col}' may not exist")
```

**执行隔离：** 通过 `threading.Timer` 设置 5 秒超时，在子线程中执行 SQLite 查询，主线程等待结果。

### 6.4 结果解读

将 SQL 查询结果（JSON 格式）传给 LLM，生成自然语言叙述。例如：
- 查询结果：`[{"region": "华东", "total": 1523000}]`
- 解读输出：「华东区域2024年销售总额约152.3万元，领先其他区域。」

## 七、MCP 工具服务与降级

**文件：** `mcp_server.py`（port 8002）、`mcp_client.py`

### 7.1 MCP 服务端点

```
POST /tools/rag_search   → 调用 RAGPipeline.query()
POST /tools/web_search   → 调用 Bocha API
POST /tools/text2sql     → 调用 Text2SQLTool.run()
```

请求/响应统一格式：
```json
{"query": "...", "params": {}, "session_id": "..."}
→ {"result": [...], "elapsed_ms": 123}
```

工具结果在 Redis 中按 `session_id` 缓存，避免同一会话重复调用相同工具。

### 7.2 ExecutorNode 降级逻辑

```python
# 优先调用 MCP 服务（HTTP）
try:
    result = mcp.call(tool_name, query, session_id=session_id)
except MCPCallError as e:
    logger.warning(f"MCP call failed: {e}, falling back to direct invocation")
    # 降级到直接调用（同进程内）
    result = _tools.dispatch(tool_name, query)
```

降级保证：即使 MCP 服务（port 8002）宕机，Agent 仍可通过直接调用继续工作。

## 八、并行执行策略

项目在三个层次使用不同并行方案：

| 层次 | 使用技术 | 场景 |
|------|---------|------|
| **DeepScout** | `asyncio.gather()` | N 个子问题 × 2 路搜索同时发起 |
| **LeadWriter** | `ThreadPoolExecutor(max_workers=6)` | 6 章节同时写作 |
| **DataAnalyst** | 顺序执行（sequential） | 图表生成依赖 SQL 结果 |

**选型理由：**
- DeepScout：I/O 密集（HTTP 请求），asyncio 无需线程开销，适合高并发 I/O
- LeadWriter：LLM API 调用受 GIL 影响小（网络 I/O），ThreadPoolExecutor 可突破 GIL 限制，且 LLM SDK 不是纯 async

**同步调用 asyncio 的技巧（从同步 LangGraph 节点发起异步操作）：**
```python
# LangGraph 节点是同步函数，无法直接 await
loop = asyncio.new_event_loop()
asyncio.set_event_loop(loop)
try:
    raw = loop.run_until_complete(_search_all(questions))
finally:
    loop.close()
```

## 九、Demo Mode 加速机制

Demo Mode 通过在各节点跳过耗时的 LLM 调用，将端到端时间从 600s+ 压缩至 ~23.5s。

| 节点 | 正常模式 | Demo Mode |
|------|---------|-----------|
| ChiefArchitect | LLM 生成6章大纲 + 5-8子问题 | 固定1章 + 1子问题（无 LLM 调用） |
| DeepScout | N 子问题并行搜索 + 事实提取 LLM | 限制1子问题，跳过事实提取 |
| DataAnalyst | SQL + LLM 解读 + matplotlib | 返回预生成 demo 图表（0.5s） |
| LeadWriter | 并行写作6章 + 摘要 LLM | 仅写2章，摘要用模板字符串 |
| CriticMaster | LLM 全面审核 | 自动通过（quality_score=0.75） |

**触发方式：** `state["demo_mode"] = True`，通过 API 请求参数 `?demo=true` 传入。

## 十、SSE 实时推送

**文件：** `api_server.py`

### 10.1 实现机制

```
研究任务启动 → 异步线程执行 Graph 2
              ↓（每个节点完成时 push 事件）
       Redis List: sse:{session_id}:events
              ↓
GET /research/stream  → StreamingResponse
                      → 每 0.5s 从 Redis LPOP 读取事件
                      → 格式化为 SSE data: {...}\n\n
                      → 推送到前端 EventSource
```

### 10.2 事件格式

```
data: {"type": "progress", "node": "deep_scout", "message": "正在搜索5个子问题..."}

data: {"type": "section_ready", "section_id": "sec_1", "title": "市场概况"}

data: {"type": "done", "quality_score": 0.82, "report_url": "/research/report/..."}
```

### 10.3 报告缓存

```python
cache_key = f"report_cache:{hashlib.md5(question.encode()).hexdigest()}"
# TTL = 3600s（1小时）
redis.setex(cache_key, 3600, json.dumps(report))

# 命中缓存时直接返回，跳过 Graph 2 全部节点（~0s vs 23.5s demo/600s full）
```
