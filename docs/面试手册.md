# 面试手册 — DeepResearch Multi-Agent 项目

> 本手册服务于 AgentProject (DeepResearch 能源行业多智能体研究系统) 的面试准备。
> 使用方式：Ctrl+F 搜索关键词 → 定位 Q 编号 → 阅读骨架回答 + 追问预演 + 陷阱提示。

---

## §0. 使用说明

### 临场搜索策略（重要）

1. **听到一个陌生关键词就立刻 Ctrl+F**，不要试图默记全文。
2. 关键词索引（§1.1）中英双列，面试官说 "multi-agent" 还是 "多智能体"都能命中。
3. 每题的 `**关键词**` 一行列出该题所有相关 term，Ctrl+F 命中后即可定位到目标题。
4. 先看 `骨架回答`（30-60 秒口播），再看 `追问预演`（常见 follow-up）。
5. 看到标 ⚠️ 的题：**先读"陷阱提示"再答**，避免触发深挖。

### 答题节奏

- 首轮 30-60 秒给骨架，**留出呼吸空间让面试官打断提问**（而不是一口气讲 3 分钟）。
- 面试官打断追问时：用"追问预演"的答案接招。
- 主动抛 1-2 个**你能展开**的方向，**引导对方在你熟悉的题上停留**。

## §2. ⚠️ 风险与说辞（最重要章节，面试前通读）

### §2.1 Gap 1: Router "针对性修复"（⚠️ 中风险）

**简历原文：**
> 经三轮迭代从 19/30 提升至 28/30，定位 Router 意图误判为系统级根因并针对性修复

**代码真相（来自 `docs/checkpoints/FINAL_CHECKPOINT.md`）：**
- 19/30 → 26/30（Round 1）：修 `verdict_kw` bug、关键词放宽、值修正、引入 `min_match` → **全部在 `tests/final_test.py` 评测框架**
- 26/30 → 25/30 → 28/30（Round 2-3）：非确定性处理、误报 forbidden 移除 → **仍在评测框架**
- S1-VDB-5 / S1-HR-5 被标 `ACCEPTABLE_PARTIAL`（~60% 通过率），Router prompt **未修改**
- "Router 被识别为根因" 真实存在 checkpoint 记录，但"针对性修复"夸大

**建议说辞（Q22 / Q23 给出完整答案）：**
- **可以说**："我们定位到 Router 在 general 意图下会误跳过 RAG 检索，这是 S1-VDB-5 / S1-HR-5 两题失败的根因。"
- **不要说**："我修改了 Router 的 prompt / 阈值 / 关键词规则。"
- **转化为工程取舍**："修 Router 会影响其他已通过的 10+ 道题，权衡后选择在评测侧用 `min_match=1` 放宽，把它降级为 `ACCEPTABLE_PARTIAL`，而不是硬改 Router。这是**局部修复 vs 全局稳定性**的权衡。"
- **被深挖"那你最终 Router 修了吗"：** 老实承认"没有，这是已知的系统局限；后续迭代规划里计划用意图 few-shot + Router 二次确认来根治。"

**高危陷阱：**
- ⚠️ 不要吹"我改了 Router 的 prompt" — 对方问 git log / 让你现场改，会穿帮。
- ⚠️ 不要说"19/30 是因为 Router 错了" — Round 0 的 19/30 主要是 `verdict_kw` bug，和 Router 无关。
- ⚠️ 不要编造"Router 准确率从 X% 提到 Y%" — 最终评测里 Router 准确率是 10/10，但那是因为评测题目本身意图明确。

### §2.2 Gap 2: Pipeline 级降级（✅ 低风险）

**简历原文：**
> Pipeline 级整个 Multi-Agent 崩溃时自动退回 legacy ReAct 保证用户可用性

**代码真相（`api_server.py:517-521`）：**

**建议说辞（Q29 给出）：**
- 正面讲功能真实，引用代码行号有底气。
- 若被问"怎么测的"，**主动承认"目前是 smoke level"**：
  > "Layer 3 验证目前是 smoke test 级别，确认 fallback 路径可达但没做故障注入。后续改进是用 `unittest.mock` patch `run_deep_research` 主动抛异常，再断言返回 state 来自 Graph 1 而非 Graph 2。这个我在 backlog 里。"

## §3. 项目专题 Q&A

## Part 1 — Multi-Agent 架构（LangGraph StateGraph）

### Q1. 为什么从单 Agent ReAct 升级到 LangGraph 多智能体？

**关键词**: multiagent, 多智能体, langgraph, stategraph, react, 单 agent 局限
**类型**: 设计决策

**骨架回答（30-60s）：**
单 Agent ReAct 用同一个 prompt 做规划+执行+反思，角色混叠导致两个问题：(1) 事实错误率高（规划时想着执行细节）；(2) 长链路下上下文爆炸，token 成本失控。我们把流程拆成 6 个专职角色——ChiefArchitect 规划、DeepScout 搜索、DataAnalyst 算数、LeadWriter 写作、CriticMaster 审核、Synthesizer 合成——每个节点只看自己那部分状态，prompt 短且专一。LangGraph 的 StateGraph 提供条件边原生支持 RE_RESEARCHING 循环，这是原生 ReAct 做不到的。

**深度展开：**
- 代码路径：`langgraph_agent.py`（双图构建）+ `backend/agents/*.py`（6 个节点各自 `run(state, llm)` 签名）
- 共享 `AgentState` TypedDict 定义在 `agent_state.py`，Part 1 字段给 ReAct 用，Part 2 字段给 Multi-Agent 用
- 每个 Agent 返回 partial state update，LangGraph 自动 merge

**可能的追问：**
- Q: 6 个角色会不会过度拆分导致调度开销比好处大？
  A: 我们测过端到端延迟 ~50s（demo 模式 49.6s），其中 LLM 调用占 >85%，节点间状态合并只占 <1s，拆分带来的专注度收益远大于调度成本。
- Q: 为什么保留了 ReAct 循环而不是全面换成多智能体？
  A: `/chat` 端点处理即时问答（单轮），多智能体 6 个节点全跑太重；`/research/report` 处理长链路报告生成才用多智能体。双图架构是用例驱动的。
- Q: LangGraph vs 自己手写一个状态机？
  A: 条件边、streaming、断点恢复是 LangGraph 开箱即用的，自己写至少 500 行且没有 ecosystem（监控、可视化）。

**陷阱提示：**
- ⚠️ 不要说"ReAct 是错的" — 我们保留了它，不能贬低。
- ⚠️ 不要说"AutoGen/CrewAI 更好所以没用" — 解释"LangGraph 的显式状态图 > AutoGen 的对话式编排，因为我们需要确定性的执行顺序"。

### Q2. LangGraph 的 StateGraph 相比 LangChain Chain 的核心优势？

**关键词**: langgraph, stategraph, langchain, chain, 条件边, conditional edge
**类型**: 设计决策 / 概念

**骨架回答：**
三点：(1) 显式状态而非隐式消息流 — AgentState TypedDict 明确声明哪些字段被谁读写；(2) 条件边支持分支和循环 — 像 RE_RESEARCHING 回到 DeepScout 这种用 Chain 做不到；(3) 原生支持 streaming 中间状态 — 我们的 SSE 推送直接订阅 `graph.stream()` 的事件。

**可能的追问：**
- Q: LangChain 也能做分支啊（RouterChain）？
  A: 能做单层分支，但嵌套循环+多路合并没有 StateGraph 清晰。我们的 ReflectorNode → PlannerNode 回边就是典型。
- Q: 为什么不用 Prefect / Airflow 这种通用 DAG？
  A: Prefect 是 batch 调度器，响应时间按秒计；Agent 场景要 ms 级状态合并，而且需要 LLM 感知的编排抽象。

### Q3. 6 个 Agent 的职责边界是怎么划分的？为什么不是 4 个或 8 个？

**关键词**: 6 agents, 职责划分, single responsibility, 多智能体粒度
**类型**: 设计决策

**骨架回答：**
按"能源行业研究报告"这个领域的自然工作流切分：从需求→搜资料→算数据→写报告→审稿→定稿。每个节点对应现实中的一个角色（架构师/情报员/数据分析师/主笔/批评家/编辑），prompt 专一且可替换。试过合并 DeepScout+DataAnalyst（都叫"数据搜集"），但两者工具不同（Web/RAG vs SQL）、输出结构不同（`raw_sources` vs `data_points`），合并后 prompt 长度翻倍、幻觉率上升，所以拆开。

**可能的追问：**
- Q: CriticMaster 和 Reflector 不是重复了吗？
  A: Reflector 是 ReAct 循环内的"下一步决策"（continue/replan/done），粒度是 step 级；CriticMaster 是 Graph 2 的报告级审核，粒度是章节级。两者不在同一个图里。
- Q: 那为什么不加 FactChecker、SourceRanker 这种更细的？
  A: 成本收益。每加一个节点多一次 LLM 调用（~5-10s 延迟 + token），我们测过 6 个节点是质量和延迟的平衡点。

### Q4. StateGraph 的 AgentState 是怎么设计的？

**关键词**: agent state, typeddict, 共享状态, 字段设计
**类型**: 设计决策

**骨架回答：**
用 TypedDict 而不是 dict 是为了 IDE 类型提示和 pydantic 兼容。字段分两组：Part 1（ReAct 用：question/intent/plan/steps_executed/reflection/confidence/final_answer/iteration/session_id）和 Part 2（Multi-Agent 用：outline/hypotheses/research_questions/facts/raw_sources/data_points/draft_sections/charts_data/references/critic_issues/pending_queries/quality_score/phase/demo_mode）。两图共用一个 state schema，transition 通过 `phase` 字段显式推进（planning→researching→analyzing→writing→reviewing→done）。

**可能的追问：**
- Q: 一个 state 共用会不会字段爆炸？
  A: 确实有 20+ 字段，但每个节点只读/写自己关心的几个，LangGraph 的 partial update 机制不会强制全量传递。
- Q: 为什么不用 pydantic BaseModel？
  A: LangGraph 0.1.x 对 TypedDict 的支持更成熟，后续版本支持 pydantic 但还没迁移。

### Q5. ChiefArchitect 的规划逻辑是怎样的？

**关键词**: chief architect, 研究规划, 大纲, 子问题拆解, hypothesis
**类型**: 设计决策

**骨架回答：**
ChiefArchitect 产出 3 个可验证假设 + 6 章能源行业标准大纲（市场/政策/竞争/技术/数据/展望）+ 5-8 个子问题供 DeepScout 并行搜索。假设要求包含具体数字或时间节点（如"2025 年光伏组件成本降至 0.7 元/W 以下"）以便后续可证伪。大纲每章带 3 个中文关键词，直接用于 Bocha 搜索和 RAG 检索。

**深度展开：**
- 代码：`backend/agents/chief_architect.py:52 run()` → LLM 返回 JSON → 验证 + 兜底 `_default_outline()`
- 兜底：LLM 返回空大纲时，fallback 到硬编码 6 章模板

**可能的追问：**
- Q: 大纲硬编码能源行业模板，换行业怎么扩展？
  A: `_default_outline()` 是兜底不是默认，正常走 LLM 生成；换行业改 `_SYSTEM` prompt 里的"章节结构"段即可。架构没绑死行业。

### Q6. DeepScout 的并行搜索是怎么实现的？

**关键词**: deep scout, 并行搜索, asyncio, gather, bocha, rag
**类型**: 设计决策

**骨架回答：**
每个子问题触发两路搜索：Bocha 网络搜索 + RAG 知识库检索。两路用 `asyncio.gather` 并行；所有子问题又用一层 `asyncio.gather` 并行。N 个子问题只用 1 个 RTT 的时间完成。搜索结果去重（URL 优先，否则 md5(snippet)），按可信度打分（RAG 基分 0.6 > Web 基分 0.5，加内容长度奖励），取 top-8 交给 LLM 抽取 3-8 条结构化事实存入 `facts`。

**深度展开：**
- 代码：`backend/agents/deep_scout.py:97 _search_all()`
- 同步→异步技巧：`asyncio.new_event_loop()` + `run_until_complete`，因为 LangGraph 节点是同步的

**可能的追问：**
- Q: 为什么 RAG 基分高于 Web？
  A: RAG 库是我们主动导入的行业文献，可信度基线比公开 Web 高。但加权方案是可调的。
- Q: Bocha 和 DuckDuckGo 为什么选 Bocha？
  A: Bocha 中文场景召回率更高，能源行业中文资料占比 > 80%。

### Q7. 哪些节点能并行跑，哪些必须串行？

**关键词**: 并行, pipeline, 串行, 节点依赖
**类型**: 设计 / 扩展

**骨架回答：**
Graph 2 主干串行：ChiefArchitect → DeepScout → DataAnalyst → LeadWriter → CriticMaster → Synthesizer。串行原因是有数据依赖：DataAnalyst 需要 DeepScout 的 research_questions，LeadWriter 需要前两者的 facts+data_points。节点**内部**有并行：DeepScout 内部 N 子问题 × 2 路搜索并行，LeadWriter 内部 6 章节 ThreadPool 并行。理论上 DeepScout 和 DataAnalyst 可以 fan-out 并行（两者互不依赖），但 LangGraph 的 fan-out 语法当时不够稳定（0.1.x），留在 backlog。

**可能的追问：**
- Q: 那怎么估 fan-out 能省多少时间？
  A: DeepScout ~4.2s (demo)，DataAnalyst ~2s，fan-out 后取 max = 4.2s，省 ~2s。实际收益有限，所以没先做。

### Q9. 既然有了多智能体，为什么还保留 5 节点 ReAct 循环？

**关键词**: react, 保留, 双图, /chat endpoint
**类型**: 设计决策

## Part 2 — 模型调度与多 Key 路由

### Q11. Router/Planner/Executor/Reflector 各自什么职责？

**关键词**: router, planner, executor, reflector, react 5 节点
**类型**: 概念 / 真实性

**骨架回答：**
Router 做意图 5 分类（policy_query / market_analysis / data_query / research / general），决定后续路径；Planner 根据意图生成 JSON 多步执行计划（最多 5 步），每步指定 action（rag_search/web_search/text2sql/doc_summary）；Executor 按计划调用 MCP 工具（失败 fallback 直接调用），把 observation 写回 state；Reflector 读 steps_executed，输出 JSON {decision: continue/replan/done, confidence, answer}，confidence ≥ 0.7 直接跳 Critic 避免过度迭代。

**可能的追问：**
- Q: 为什么 Planner 限最多 5 步？
  A: 经验值。超过 5 步链路太长，token 成本和错误累积都不划算；若不够就触发 replan 而不是加步数。
- Q: Reflector 的 confidence 怎么来的？
  A: LLM 自评给出的 0-1 浮点数，结合 `_REFLECTOR_SYSTEM_V2` prompt 要求。不完全可信但作为启发式足够。

### Q12. 为什么要为每个 Agent 分配不同的 API Key？

**关键词**: llm_router, 多 key, rate limit, 速率限制, 并发
**类型**: 设计决策

**骨架回答：**
单 Key 并发会触发 provider 的 rate limit（每分钟请求数/token 数上限）。LeadWriter 内部 6 章节并行写作，如果全用同一个 Key，大概率触发 429。为每个 Agent 角色分配独立 Key 把并发打散到不同配额池。具体分配在 `llm_router.py`：router+chief 用 KEY_1（低并发）、deep_scout 用 KEY_2、data_analyst 用 KEY_3、lead_writer 用 KEY_4（最高并发）、critic 用 KEY_5、synthesizer 用 KEY_6。

**可能的追问：**
- Q: 如果只有 1 个 Key 呢？
  A: FALLBACK_KEY 机制会让所有 role 回退到 KEY_1，功能不崩但会退化到串行并发，LeadWriter 可能触发 429 重试。
- Q: 为什么不做动态负载均衡？
  A: 目前是静态分配，没做跨 Key 的实时容量感知（哪个 Key 当前空闲就用哪个）。是 backlog，后续可以接 token bucket。

---

### Q13. 降级链 `KEY_N → KEY_1 → OPENAI_API_KEY` 的设计思想？

**关键词**: fallback chain, 降级, key 缺失
**类型**: 设计决策 / 边界

**骨架回答：**
三级降级：(1) 角色专属 key 未配置 → 退到 KEY_1；(2) KEY_1 未配置 → 退到通用 OPENAI_API_KEY；(3) 全部未配置 → 返回 empty-key client，调用时报错但不崩。保证**部署灵活性**：本地开发只有 1 个 key 也能跑，生产再配多 key 性能更好，代码不用改。

---

### Q14. 如何决定哪个 Agent 用大模型哪个用小模型？

**关键词**: model scheduling, 模型调度, 大小模型, 推理 vs 分类
**类型**: 设计决策

**骨架回答：**
按任务复杂度切分（通过 `ROLE_TO_MODEL_ENV` 配置）：
- **小模型（分类/抽取类）**：Router（意图 5 分类）、DeepScout（事实抽取）
- **大模型（推理/写作类）**：Planner（多步规划）、LeadWriter（长文写作）、CriticMaster（对抗式审核）、Synthesizer（报告合成）
- **中等模型**：DataAnalyst（SQL 生成 + 结果解读）

这么做省 token、降延迟。Router 用 MiniMax-M2.5-small 比用大模型快 ~3 倍，准确率差不多（因为意图分类任务简单）。

### Q15. DeepScout 高并发下如何避免 rate limit？

**关键词**: deep scout, 并发, asyncio, rate limit, 重试
**类型**: 边界 / 实现

**骨架回答：**
DeepScout 本身**不直接调 LLM**（搜索是 HTTP 请求 Bocha/RAG），真正调 LLM 的是事实抽取（`_extract_facts`）只 1 次。所以它的主要瓶颈是 HTTP 并发不是 LLM rate limit。HTTP 层面用 `requests` + asyncio `run_in_executor`，每个请求 20s 超时，失败返回空列表不抛异常——保证单个子问题挂掉不影响其他子问题。

## Part 3 — 质量保障闭环（CriticMaster + 评测）

### Q16. CriticMaster 为什么识别这 6 类问题（hallucination/missing_source/logic_error/outdated/incomplete/bias）？

**关键词**: critic master, 6 类问题, hallucination, 审核
**类型**: 设计决策

**骨架回答：**
这 6 类是行业研究报告的典型缺陷，参考了 Anthropic Constitutional AI 的"批评维度"和研究报告写作指南。hallucination/missing_source 是 RAG 系统的高频病（LLM 编数字）；logic_error/bias 是推理型错误；outdated/incomplete 是覆盖度错误。6 类足以覆盖 >90% 的质量问题；再细分（如"口语化"）会增加 CriticMaster 自己的幻觉风险。

### Q17. quality_score 的评分标准从哪来？主观还是客观？

**关键词**: quality score, 评分, 主观
**类型**: 真实性 / 设计

**骨架回答：**
**主观为主，量化锚定**。评分区间和语义锚点（0.9+ 优秀、0.7-0.9 良好、0.5-0.7 一般、<0.5 不合格）写在 CriticMaster 的 system prompt 里，让 LLM 按标准打分。我们没有真 ground truth 校准（那需要人工标注），所以承认它是 LLM-as-judge 的置信度 proxy。用它做**相对比较**（同一问题两个版本的分数）比绝对值更有意义。

**陷阱提示：**
- ⚠️ 不要说"quality_score 是客观指标" — 对方追问"和人工标注对齐率多少"你答不上。

### Q18. RE_RESEARCHING 阈值从 0.75 调到 0.6，为什么？

**关键词**: re_researching, 阈值, 0.6, 收敛
**类型**: 真实性 / 设计

**骨架回答：**
0.75 太严格，导致频繁触发重搜循环，端到端延迟翻倍。实测 quality_score ≈ 0.7 的报告质量人工看已经 acceptable，把阈值降到 0.6 后触发率从 ~40% 降到 ~10%，端到端延迟稳定在 50s 左右。这是**质量 vs 延迟的取舍**。

**深度展开：**
- 代码：`backend/agents/critic_master.py` — `pending_queries and quality_score < 0.6`
- 另外加了 iteration-aware 收敛：iteration ≥ 2 强制 done，防止无限循环

### Q19. 为什么 RE_RESEARCHING 最多循环 2 次？

**关键词**: re_researching, max iteration, 收敛保护
**类型**: 边界 / 设计

**骨架回答：**
防"完美主义死循环"。LLM 的 CriticMaster 是 adversarial 模式，总能挑出问题；如果无上限，能一直循环下去。实测 2 次后再循环的边际收益很小（quality_score 提升 < 0.05），但成本翻倍。2 次是经验值。

### Q20. 怎么防止 CriticMaster 自己产生幻觉（误判）？

**关键词**: critic, 误判, 幻觉, adversarial
**类型**: 边界

**骨架回答：**
三层防护：(1) CriticMaster 的 prompt 强制要求输出**具体的问题描述+严重程度+修复查询词**，不能只说"有问题"；(2) `severity in (high,medium)` 且有 `fix_query` 才会触发 pending_queries，过滤掉"水"的批评；(3) 整体由 iteration ≤ 2 兜底，就算误判也不会死循环。

## Part 4 — MCP 工具层与三层降级

### Q24. 为什么把工具封装成独立的 MCP 服务进程？

**关键词**: mcp, model context protocol, 独立进程, 工具服务
**类型**: 设计决策

**骨架回答：**
三点：(1) **解耦** — 主 API 服务（8003）和工具服务（8002）独立部署，RAG/SQL 挂了不影响对话；(2) **可扩展** — 多个 Agent 实例可以共享同一个 MCP 服务，避免每个 Agent 都加载一份 BGE-m3 模型（~2GB）；(3) **标准化** — MCP 协议是 Anthropic 推的 Agent 工具交互标准，后续可接入 Claude Desktop 等生态。

---

### Q25. 三层降级的"工具级" 是什么？

**关键词**: tool level, 工具级降级, try except, isolation
**类型**: 设计 / 边界

**骨架回答：**
每个工具端点（`mcp_server.py` 的 `/tools/rag_search` 等）独立 try/except，返回 `ToolResponse(result=..., error=...)` HTTP 200。这样**单工具故障不污染其他工具**：Milvus 挂了 rag_search 返回 error，但 text2sql、web_search 仍然可用。失败返回 200 而非 500 是有意为之——让 MCPClient 能统一解析而非做 HTTP status 兜底。

---

### Q26. MCPClient 层的封装细节是什么？

**关键词**: mcpclient, mcpcallerror, 统一异常, per-tool timeout
**类型**: 实现 / 真实性

**骨架回答：**
`mcp_client.py:20-62` MCPClient 类：(1) 每个工具独立 timeout（rag/web/doc_summary 各 30s，text2sql 60s，因为 SQL 生成+执行慢）；(2) 网络层异常（Connection/Timeout）和业务层异常（`response.error != null`）统一封装成 `MCPCallError(tool, msg)`——调用方只 catch 一个异常；(3) 用 `requests.Session` 复用连接；(4) 抛异常而不是返回 None，让调用方 fallback 逻辑清晰。

## Part 5 — MemGPT 双层记忆

### Q30. 什么是 MemGPT 双层记忆？你的实现和原论文有什么差异？

**关键词**: memgpt, 双层记忆, core memory, archival memory, 差异
**类型**: 概念 / 真实性

**骨架回答：**
MemGPT 原论文（Packer 等 2023）的核心思想：LLM 上下文有限，分两层——Core Memory 常驻 prompt（persona + human 信息），Archival Memory 存外部向量库按需检索。我的实现：Core Memory 用 Redis（per-session key），Archival Memory 用 Milvus（BGE-m3 嵌入）。**与原论文差异**：(1) 原论文用 LLM 自主触发 `core_memory_append/archival_memory_search` 等工具调用，我实现了部分——Reflector 节点后 LLM 判断是否写入/检索，但工具数比论文少；(2) 原论文有 FIFO Queue 作为工作记忆，我没实现（我们的 session 较短，用 steps_executed 代替）；(3) 原论文强调 self-editing，我实现了 `core_memory_replace`，但没实现复杂的自编辑循环。

### Q31. Core Memory 为什么用 Redis 而不是进程内字典？

**关键词**: core memory, redis, session, 持久化
**类型**: 设计决策

**骨架回答：**
三点：(1) **多实例共享** — API 服务横向扩展后多进程共享 session，字典做不到；(2) **TTL 管理** — Redis setex 原生支持 session 过期（3600s），字典要手写过期；(3) **跨进程** — MCP 服务进程也要读 core memory（memory-guided SQL 场景），Redis 作为共享层最自然。

---

### Q32. Archival Memory 为什么用 Milvus 而不是 SQLite / PostgreSQL pgvector？

**关键词**: milvus, archival memory, pgvector, 向量数据库选型
**类型**: 设计决策

**骨架回答：**
Archival 的检索是**语义相似度**，需要 ANN（近似最近邻）索引，不是精确匹配。pgvector 可以但当时 PG 生态我们没搭；SQLite 没有向量索引支持。选 Milvus 是因为：(1) RAG 已经用 Milvus，复用集群成本低；(2) 支持 IVF_FLAT/HNSW/FLAT 多种索引可按数据量切换；(3) BGE-m3 1024 维向量原生支持。archival_memory 集合用 FLAT 索引（数据量小 < 千条，FLAT 精确检索更合适）。

### Q33. BGE-m3 模型在 RAG 和 Archival 里复用，怎么做的？

**关键词**: bge-m3, 复用, singleton, embedding
**类型**: 实现 / 设计

**骨架回答：**
`MemGPTMemory(rag=_rag)` 构造时传入已加载的 RAGPipeline，复用 `rag.embed` 方法。如果不传（`rag=None`）会自己 load 一份 BGE-m3 ~2GB 内存 + ~15s 启动时间。生产环境必须复用，否则进程内存爆炸。

### Q34. Core Memory 的 2000 字符上限怎么定的？超了怎么处理？

**关键词**: core memory, 2000, fifo, 字符上限
**类型**: 设计 / 边界

**骨架回答：**
2000 字符对应 ~500 token 英文 / ~1500 token 中文，大约占上下文窗口 (32k-128k) 的 1-5%，经验安全上限。超了用 FIFO 策略——最早的内容先出（保留最新用户信息）。实际在 test_section2_memory.py 的 MEM-5 回归测试里验证：写超过 2000 后断言总长 ≤ 2000 且最新内容保留、最旧被截断。

### Q35. Reflector 节点怎么决定是否写入记忆？

**关键词**: reflector, 记忆判断, llm 决策, memory injection
**类型**: 实现

**骨架回答：**
Reflector 输出 JSON 时包含可选的 `memory_actions` 字段（`core_memory_append` / `core_memory_replace` / `archival_memory_insert` / `archival_memory_search`）。LLM 根据 steps_executed 内容判断：用户提到个人背景（职位/公司）→ 触发 `core_memory_append`；产生值得长期保留的数据结论 → 触发 `archival_memory_insert`；当前问题需要历史上下文 → 触发 `archival_memory_search`。关键：**LLM 主动触发而不是硬规则**，避免无关信息污染记忆。

### Q36. 跨 session 的 Archival 检索如何保证隔离？

**关键词**: archival, cross-session, 隔离, user scope
**类型**: 边界 / 扩展

**骨架回答：**
archival_memory 集合有 `session_id` 字段但检索时**不按 session_id 过滤**——这是设计意图：Archival 存"跨会话有价值的结论"（数据分析结果、用户偏好），新 session 也应能检索。**用户级隔离**还没做（多用户场景 user_id 维度），目前是单租户假设。多租户扩展是给 session_id 升级为 user_id，并在 `_archival.search` 的 expr 里加 `user_id == '{uid}'`。

### Q37. 记忆回归测试覆盖了哪些场景？

**关键词**: memory regression, mem-1, mem-6, 回归测试
**类型**: 真实性

**骨架回答：**
`tests/test_section2_memory.py` 里 6 个用例：
- **MEM-1** core_memory_write：首次写入是否触发，断言 persona/human 中含关键词
- **MEM-2** archival_insert：会话内插入 archival，断言 entity count before < after
- **MEM-3** archival_search：跨 session 检索，断言 top-1 score ≥ 0.5
- **MEM-4** core_memory_update：`core_memory_replace` 是否更新同 session 内容
- **MEM-5** fifo_cap：写超过 2000 后 FIFO 截断，断言最新保留+最旧截掉
- **MEM-6** memory_injection：注入的 core memory 是否影响 Planner 工具选择

## Part 6 — RAG 基础与检索优化

### Q38. 为什么选 BGE-m3 而不是 OpenAI ada-002？

**关键词**: bge-m3, ada-002, embedding 选型, 中文
**类型**: 设计决策

**骨架回答：**
四点：(1) **中文场景更强** — BGE-m3 在 C-MTEB 中文基准上排名靠前，能源行业中文资料占比 >80%；(2) **本地部署** — ada-002 要调 OpenAI API，每次检索都是 HTTP call + 费用，BGE-m3 本地 GPU 推理 <50ms；(3) **维度合适** — 1024 维比 ada-002 的 1536 维轻，Milvus 索引更快；(4) **多语言** — BGE-m3 原生支持 100+ 语言，未来扩展方便。

---

### Q39. 为什么 RAG 用 IVF_FLAT 而 Archival 用 FLAT？

**关键词**: ivf_flat, flat, hnsw, 索引选型
**类型**: 设计决策

**骨架回答：**
按数据规模选。RAG 主库 ~10k-100k 向量用 IVF_FLAT（COSINE + nprobe=16），比暴力搜索快 5-10x 且精度几乎无损。Archival 只有 ~几百到几千条向量，FLAT 暴力搜索毫秒级就够，上 IVF 反而因分簇开销变慢。**没用 HNSW** 是因为：HNSW 构建慢且内存占用大，小数据量（< 100万）边际收益不明显；百万级以上才建议 HNSW。

---

### Q40. chunk 512 tokens + 50 overlap 是怎么调的？

**关键词**: chunking, 512, overlap, 分块参数
**类型**: 真实性 / 设计

**骨架回答：**
512 tokens 对应约 ~400 中文字符，是段落的自然粒度（一个完整段落平均 200-600 字符），保证**语义完整**；50 token overlap（~40 字符）处理**跨段落查询**——如"X 的优点有哪些"可能答案跨两个段落。值是经验参数，没做 exhaustive 网格搜索。小了（256）召回率高但精度降；大了（1024）单 chunk 噪声大。

---

### Q41. SHA-256 chunk 去重的局限性？

**关键词**: sha-256, dedup, chunk 去重, 版本管理
**类型**: 边界

**骨架回答：**
基于 chunk 内容 hash，局限：(1) **版本管理失效** — 同文档更新一小部分，大部分 chunk 未变会被判重复，但旧 chunk 仍在库里（没主动删除）；(2) **近似重复识别不了** — 只改一个标点的 chunk hash 不同视为新 chunk；(3) **跨文档重复** — 多个来源有相同段落只保留一份，丢失来源信息。解决方案：引入 document-level versioning（source+updated_at 复合键），或用 SimHash 替代精确 hash。

---

### Q42. score_threshold = 0.45 怎么定的？

**关键词**: score threshold, 0.45, 检索阈值
**类型**: 真实性 / 设计

**骨架回答：**
经验值。BGE-m3 的 COSINE 相似度分布：0.3 以下基本无关，0.4-0.5 部分相关，0.5+ 强相关。0.45 是宁可多召回少精度的选择——Ranker 在下游（CriticMaster）能过滤低质结果。更高的阈值（如 0.6）会漏召（unanswerable 题本来库里就没答案，高阈值全过滤掉了反而看不出"无答案"的信号）。

---

### Q43. mAP@10 从 0.58 提升到 0.82 是怎么做到的？

**关键词**: map@10, 0.58, 0.82, bm25, rrf, hybrid retrieval, rerank
**类型**: 真实性 / 设计

**骨架回答：**
从 Dense 单路径升级到 **Dense + BM25 融合**（来自 `docs/troubleshooting-log/issue-20260407-001.md`）：
- Dense only（BGE-m3）：mAP@10 = 0.58 — 语义好但关键词不匹配时漏召
- + BM25：先独立跑两路检索各取 top-10
- + **RRF (Reciprocal Rank Fusion)** 合并两路排序：`score = Σ 1/(k+rank_i)`，k=60
- 最终：mAP@10 = 0.82（+41%）
- 延迟代价：180ms → 210ms（+17%）可接受

核心洞察：Dense 捕捉语义，BM25 捕捉精确关键词（如产品型号、版本号），融合互补。

## Part 7 — Text2SQL

### Q44. Text2SQL 的三步管道为什么这么设计？

**关键词**: text2sql, 三步管道, ambiguity, 歧义消解, term_dict
**类型**: 设计决策

**骨架回答：**
用户问题→SQL 直接一步转太脆弱。拆三步：(1) **歧义消解** — LLM 把业务术语映射到 SQL 表达式（"营收"→`SUM(amount)`、"均价"→`AVG(unit_price)`），`term_dict` 给 few-shot；(2) **SQL 生成** — 带着消解后的上下文 + schema 生成 SELECT；(3) **结果解读** — SQL 结果 JSON 传给 LLM 转自然语言。三步隔离让每步的 prompt 短、可独立评测。

---

### Q45. 只允许 SELECT 够防 SQL 注入吗？

**关键词**: sql injection, select 白名单, nested query, 大表扫描
**类型**: 边界

**骨架回答：**
够防"删表/改数据"（DDL/DML 白名单拒绝），但**不够防滥用**：(1) 深嵌套子查询可能资源爆炸——代码里有 nesting depth 检查；(2) 无 WHERE 的全表 SELECT——强制自动 append `LIMIT 50`；(3) 列名不存在——生成后做 schema 比对警告（不阻塞）；(4) 执行层用 SQLite read-only mode 连接 + threading 5s 超时。多层防护组合。完全安全需要沙箱进程隔离，目前没做。

---

### Q46. term_dict 怎么维护？新业务术语怎么加？

**关键词**: term_dict, 业务术语, 可维护性
**类型**: 扩展

**骨架回答：**
term_dict 在 `backend/tools/text2sql_tool.py` 里是 Python dict，也在 `resources/data/schema_metadata.json` 里有 JSON 版本。加术语只改 JSON 然后重启服务（生产环境可接 Redis 做热更新，目前没做）。新术语测试：加一条 few-shot 用例到 `test_text2sql.py` 验证生成的 SQL 正确。

### Q47. 5 秒 SQL 执行超时太严格？复杂 SQL 被误杀怎么办？

**关键词**: sql timeout, 5s, 误杀
**类型**: 边界

**骨架回答：**
5s 是 SQLite 本地执行的上限，真实复杂 SQL 也是毫秒级（库只有 sales + products 两表，< 10k 行）。**误杀是功能而非 bug**——如果 LLM 生成了 5s 还跑不完的 SQL，大概率是生成错了（如笛卡尔积）或数据库出问题了，超时让上层 fallback 比等更好。扩展到大数据后超时要改成可配置 + 查询 cost estimation。

## Part 8 — 并发与性能

### Q48. DeepScout 用 asyncio，LeadWriter 用 ThreadPoolExecutor，为什么不统一？

**关键词**: asyncio, threadpoolexecutor, 并发选型, gil
**类型**: 设计决策

**骨架回答：**
按任务性质选。DeepScout 是**纯 I/O 密集**（HTTP 请求 Bocha/RAG），asyncio 单线程事件循环最轻量，无线程创建开销；LeadWriter 调 LLM SDK（OpenAI/MiniMax），**SDK 本身不是 async 原生**——虽然是 I/O 密集但必须走同步调用，用 ThreadPool 突破 GIL（GIL 在 I/O 等待时释放）。统一用 asyncio 的话 LLM SDK 要 `run_in_executor` 套一层，本质还是线程池，不如直接用 ThreadPool 语义清晰。

---

### Q49. max_workers = 6 怎么定的？调大更快吗？

**关键词**: max_workers, 6, 线程池大小
**类型**: 设计

**骨架回答：**
6 等于最大章节数（CheapArchitect 默认生成 6 章）。再大意义不大——章节更多时没有更多任务并行；更小会串行化损失并行收益。如果 rate limit 够高理论上可以任务级并发更高，但 LLM_KEY_4 的配额只能撑 6 个并发请求，大了会 429。

---

### Q50. 从同步 LangGraph 节点启动 asyncio 的技巧？

**关键词**: asyncio, sync to async, event loop, run_until_complete
**类型**: 实现

**骨架回答：**
LangGraph 节点是同步函数，不能直接 await。用 `asyncio.new_event_loop()` + `run_until_complete` 模式：
```python
loop = asyncio.new_event_loop()
asyncio.set_event_loop(loop)
try:
    raw = loop.run_until_complete(_search_all(questions))
finally:
    loop.close()
```
每次都 new 新 loop 避免嵌套 loop 冲突（Python 3.10+ 对 nested loop 严格）。

### Q51. Demo Mode 从 263.9s 压到 49.6s，具体怎么做的？

**关键词**: demo mode, 263.9s, 49.6s, 81%, 优化
**类型**: 真实性

**骨架回答：**
7 个节点层面的优化（来自 issue-20260413-005）：
- **ChiefArchitect**：跳过 LLM，固定返回 1 章大纲 + 1 子问题（省 ~15s）
- **DeepScout**：限制到 1 子问题，跳过 fact extraction 的 LLM 调用（省 ~20s）
- **DataAnalyst**：跳过 SQL，直接返回预生成 demo 图表（matplotlib 2s）
- **LeadWriter**：只写 2 章（而非 6），跳过 summary 的 LLM 调用（省 ~19s）
- **CriticMaster**：跳过审核 LLM，auto-pass quality_score=0.75（省 ~15s）
- **Synthesizer**：跳过 revision 的 LLM 调用
- 加了 **report_cache**：md5(question) 命中直接返回（~1ms vs 49.6s）

关键洞察：demo 模式不是为了性能优化生产逻辑，是为了**让前端有可展示的 smoke test**。真实模式延迟 ~60s 是用户可接受的。

### Q52. 真实模式下端到端多长时间？哪个节点最慢？

**关键词**: 真实模式, 延迟, 瓶颈节点, e2e latency
**类型**: 真实性

**骨架回答：**
真实模式 ~50-60s（LLM-bound）。最慢节点是 **LeadWriter**——6 章并行但每章是长文生成（500-800 字），单次 LLM call ~10-15s。虽然并行，但 rate limit 会让部分章节排队，加上 summary LLM call 总时间 ~20-30s。其次慢是 CriticMaster（对抗式审核 LLM ~10s）。RAG 检索 + Bocha 搜索加起来 < 5s 不是瓶颈。

## Part 9 — 工程化与扩展性

### Q53. 横向扩展到多实例，session 状态怎么路由？

**关键词**: 横向扩展, session affinity, sticky session
**类型**: 扩展

**骨架回答：**
当前 session 状态全在 Redis（per-session key），**天然支持多实例**——负载均衡器可以随机分发请求，任何实例都能读到同一个 session 的 core memory 和 steps。唯一要注意是 SSE 连接要 sticky（同一 session 的 EventSource 流要回到同一实例，否则 Redis LPUSH 推不到正确的 socket）。用 nginx ip_hash 或 session cookie 做 sticky 即可。

### Q54. 单点故障点在哪？如果 Redis 挂了会怎样？

**关键词**: spof, 单点故障, redis, milvus
**类型**: 边界

**骨架回答：**
- **Redis 挂**：Core Memory 读失败 → Planner prompt 没有用户上下文，但不崩；session steps 写失败 → 当次请求降级到无记忆模式。生产用 Redis Sentinel/Cluster 解决。
- **Milvus 挂**：RAG 检索全挂，ExecutorNode 会 catch 异常返回空结果，Planner 会 replan 或放弃 RAG 走 Web。Archival 读失败不影响主流程。
- **MCP 服务挂（port 8002）**：MCPClient 抛 MCPCallError，ExecutorNode fallback 到直接调用 `_tools`（同进程内），完全可用。
- **LLM API 挂**：没有兜底（没法本地跑 LLM），整个 Agent 不可用。这是真正的 SPOF，需要多 provider 切换（目前多 Key 但同 provider）。

## Part 10 — SSE 与实时推送

### Q58. SSE vs WebSocket 为什么选 SSE？

**关键词**: sse, websocket, server-sent events, streamingresponse
**类型**: 设计决策

**骨架回答：**
单向推送场景 SSE 够用且更简单：(1) **浏览器原生 EventSource API**，前端 3 行代码接入；(2) **基于 HTTP**，nginx/CDN 原生支持，不需要特殊代理配置；(3) **自动重连**——EventSource 断连后自动重试，WebSocket 要手写重连逻辑。WebSocket 优势（双向、二进制）对我们没用——用户发消息用 REST POST 足够。

---

### Q59. Redis List 作事件队列，客户端断开后事件堆积怎么办？

**关键词**: sse, 事件堆积, 客户端断开, ttl
**类型**: 边界

**骨架回答：**
两个机制：(1) `sse_events:{sid}` key 随 session TTL 一起过期（3600s）自动清理；(2) SSE 流有 600s 超时——超过不再消费，后端 pipeline 正常推到 Redis 直到 `done` 事件，Redis List 过期时一起丢。改进空间：用 Redis Streams（带消费者 offset）比 List 更合适，支持断线重连读历史事件。目前 List + TTL 是简化实现。

### Q60. report_cache 用 md5(question) 做 key，合理吗？

**关键词**: report cache, md5, 缓存粒度
**类型**: 边界 / 扩展

**骨架回答：**
合理但粗糙：(1) **相同问题共享缓存** ✅ 高频问题省 50s+；(2) **忽略了用户上下文**——同一问题不同用户记忆不同应得到不同答案，当前缓存命中会返回和用户记忆无关的版本；(3) **忽略了时效性**——新闻类问题一小时后数据可能过期。改进：key 升级为 `md5(user_id + question + int(time/3600))`，加用户维度 + 小时级时间桶。目前是 MVP 妥协。

### Q61. ReAct 论文的核心思想？和你项目里的 ReAct 是什么关系？

**关键词**: react, reasoning, acting, yao 2022

**骨架回答：**
ReAct（Yao et al. 2022）提出 **Reasoning 和 Acting 交错**的模式：LLM 在每步输出 `Thought → Action → Observation`，用外部工具补齐知识、用思考引导下一步行动。对比纯 CoT（只推理不查）、纯 Action（只调工具不反思），ReAct 在 HotpotQA / ALFWorld 上 accuracy 和可解释性都更优。

**项目里的体现：**
Graph 1（`langgraph_agent.py` ReAct 子图）就是经典 ReAct——5 节点 Router/Planner/Executor/Reflector/Critic 对应 Plan → Act → Observe → Reflect → Verify。confidence≥0.7 提前退出对应论文里的 "stop when evidence sufficient"，MAX_ITER=3 对应 trajectory length cap。

**可能的追问：**
- Q: ReAct 最大的缺点？→ A: **错误累积**——前一步 observation 错了后面越走越偏，我们用 Reflector 做每步 self-check 部分缓解。
- Q: Plan-and-Execute 和 ReAct 区别？→ A: Plan-and-Execute 先一次性出完整 plan 再逐步执行（更少 token、更少 loop），ReAct 每步都重新推理（更灵活、更贵）。

---

### Q62. MemGPT 论文的核心贡献？你的实现和论文有什么差异？

**关键词**: memgpt, virtual context, self-editing, packer 2023

**骨架回答：**
MemGPT（Packer et al. 2023）把 LLM 有限上下文窗口类比成**操作系统虚拟内存**：Main Context（RAM，in-context）+ External Context（Disk，out-of-context），LLM 通过 function call 自主决定调入/调出（`core_memory_append`, `archival_memory_insert/search`）。核心创新是 **self-editing memory**——不是 RAG 把检索硬塞进 context，而是 LLM 自己决定什么时候读写哪层。

**项目 vs 论文的差异（重要！主动说出来）：**
1. **论文全自主，项目半自主**——论文里 LLM 完全决定何时调 archival_insert，我们是在 Planner 节点**规则触发**（query 含"记住/my name"时 append；每个会话结束自动 archival_insert）。原因：MiniMax-M2.5 的 function calling 稳定性不如 GPT-4，完全自主会漏写记忆。
2. **论文有 recursive summarization，项目是 FIFO**——我们的 Core Memory 满了（2000 字符）直接丢最老的句子，论文是让 LLM 做摘要压缩。原因：摘要压缩需要额外 LLM call，MVP 阶段做了简化。
3. **论文只有一份 Main Context，项目分 persona + human 两块**——我们把 persona 固定不淘汰、human 才做 FIFO，防止 Agent 人设被冲走。

**陷阱提示** ⚠️：
不要说"我完全复现了 MemGPT"——面试官追问"那 recursive summarization 怎么做的"会穿帮。坦诚讲差异反而加分。

### Q63. LangChain / LangGraph / LlamaIndex / AutoGen / CrewAI 怎么选？

**关键词**: langchain, langgraph, llamaindex, autogen, crewai

**骨架回答：**
- **LangChain**：链式 LCEL，适合**线性 pipeline**（RAG / chain-of-tools）。缺点：复杂控制流难表达。
- **LangGraph**（本项目选型）：基于 StateGraph 的**图式编排**，支持条件边、循环、状态共享——适合 Multi-Agent。
- **LlamaIndex**：偏 RAG 场景，文档索引/分块/重排做得比 LangChain 细；但编排能力弱。
- **AutoGen**（微软）：**多 Agent 对话**范式，Agent 之间通过自然语言 message 沟通，适合代码生成/辩论。
- **CrewAI**：角色化 Multi-Agent（Researcher / Writer / Reviewer），prompt 层封装得好，但底层仍是 LangChain。

**为什么选 LangGraph：** 我们需要 (1) 条件分支（Critic 决定通过还是 RE_RESEARCHING），(2) 循环（最多迭代 2 次），(3) 状态跨 Agent 共享（AgentState 贯穿 7 个节点）——这三个需求 AutoGen 的对话范式不方便表达、LangChain 的链式太线性。LangGraph 的图抽象最匹配。

**可能的追问：**
- Q: 为什么不用 CrewAI？→ A: CrewAI 的 crew/task/agent 抽象够用但**状态管理弱**，我们 AgentState 里带 critic_issues/iteration 跨节点同步用 LangGraph reducer 更顺手。
- Q: 会不会锁死在 LangGraph 生态？→ A: AgentState 是 pydantic model、Agent 都是纯函数，迁移到 AutoGen 只需重写编排层，不到 1 天。

### Q64. Anthropic 2024 博客里 Agent vs Workflow 的区分是什么？你的项目属于哪种？

**关键词**: agent vs workflow, anthropic, orchestration

**骨架回答：**
Anthropic 2024 年 "Building Effective Agents" 博客定义：
- **Workflow**：LLM 和工具通过**预定义代码路径**编排，控制流在代码里写死。
- **Agent**：LLM **动态指挥**自己的流程和工具使用，控制流由 LLM 在运行时决定。

**项目实际是混合体：**
- Graph 1（ReAct 子图）偏 Agent——Executor 由 LLM 决定下一步 action。
- Graph 2（Multi-Agent 主图）偏 Workflow——ChiefArchitect → DeepScout → ... 的顺序是代码写死的 StateGraph edge。
- **唯一 Agent 式的决策点**：CriticMaster 输出 `pass / revise / re_research` 决定走哪条条件边。

**为什么这样选：** 纯 Agent 灵活但**不可预测**（token 成本失控、Agent 可能死循环）；纯 Workflow 便宜可控但不智能。工业场景下**静态 Workflow + 少数关键决策点 Agent 化**是稳定性和能力的最佳权衡。

### Q65. Prompt Engineering 的主流范式？你在项目里用了哪些？

**关键词**: few-shot, cot, tot, self-consistency, prompt engineering

**骨架回答：**
主流范式：
1. **Zero-shot**：直接问
2. **Few-shot**：给 3-5 个示例
3. **CoT（Chain-of-Thought）**："Let's think step by step"，显式展开推理
4. **Self-Consistency**：同 prompt 采样 N 次投票
5. **ToT（Tree-of-Thoughts）**：LLM 树搜索多条推理路径
6. **ReAct**：Thought + Action 交错

**项目里用了：**
- **Few-shot**：Router 的意图分类 prompt 给了 data_query/analysis/research/general 各 2 个示例
- **CoT 隐式版**：Planner 输出 `think: ... plan: [...]` 让模型先推理再给 plan
- **ReAct**：Graph 1 全盘用
- **结构化输出**：所有 Agent 都用 JSON schema 约束输出（`json.loads` 解析失败会 retry），这个不算 prompt engineering 但配套

**没用的原因：**
- Self-Consistency——N 次采样成本 N 倍，不符合 demo 49.6s 的性能目标
- ToT——过度设计，我们的问题不需要这个复杂度

### Q66. LLM 幻觉的成因和缓解手段？你项目里怎么做？

**关键词**: hallucination, groundedness, faithfulness, rag

**骨架回答：**
成因：
1. **Parametric memory 过时**（训练数据 cutoff）
2. **分布外问题**（训练时没见过）
3. **上下文误导**（检索到错误文档 garbage in garbage out）
4. **采样随机性**（temperature > 0 时）

缓解：
1. **RAG 把检索文档塞进 context**——grounding
2. **引用 citation 要求**（我们 Synthesizer 强制要求带 [Source N]）
3. **多路校验**（我们 CriticMaster 独立 LLM 审核，相当于 secondary check）
4. **置信度阈值**（Reflector confidence < 0.7 触发 RE_RESEARCHING）
5. **Temperature 调低**（LeadWriter 用 0.3 而非 0.7）

**项目的三道防线：**
- 第一道：RAG 检索用 score 阈值 0.45 过滤低相关文档，减少 garbage in
- 第二道：DeepScout 并行多源（RAG + Web + Text2SQL）交叉验证
- 第三道：CriticMaster quality_score < 0.6 打回 RE_RESEARCHING

**追问预演：**
- Q: 还是会有幻觉怎么办？→ A: CriticMaster 的 hallucinated 字段会标记"结论无来源支撑"，revise 路径会强制 Synthesizer 按 critic_issues 改写，最多 2 次。
- Q: 能完全消除吗？→ A: 不能，只能降低概率。工业做法是**端到端评测**（groundedness 指标）持续监控。

### Q67. RAG pipeline 的经典优化手段？你用了哪些？

**关键词**: rag, hyde, rag-fusion, rerank, hybrid search

**骨架回答：**
优化手段（按阶段）：
1. **查询侧**：HyDE（生成假设性答案再 embed）、Query Rewrite、Multi-Query
2. **检索侧**：Hybrid Search（dense + BM25 + RRF 融合）、父子 chunk
3. **重排侧**：Cross-Encoder Rerank（bge-reranker）、LLM Rerank
4. **生成侧**：Context Compression（LLMLingua）、去重

**项目用了：**
- Paragraph 级分块（512 tokens, 50 overlap）而非固定长度
- SHA-256 dedup 防重复 chunk
- score 阈值 0.45 过滤
- **Hybrid Search（Dense + BM25 + RRF）**——这是项目里 mAP@10 从 **0.58 → 0.82（+41%）**的关键优化，对应 Bullet 的"检索优化"

**没用的原因：**
- HyDE：延迟 +1s，demo 模式无法接受
- Rerank：bge-reranker 需要额外 GPU 吞吐，资源约束下没上

**追问预演：**
- Q: 0.58 → 0.82 怎么测的？→ A: 人工标注 30 条 query 的 ground truth passage，算 mAP@10。
- Q: RRF 的公式？→ A: `score(d) = Σ 1/(k + rank_i(d))`，k=60 经验值。

### Q68. Function Calling 和 Tool Use 的 API 差异？

**关键词**: function calling, tool use, openai, anthropic

**骨架回答：**
- **OpenAI Function Calling（2023 初）**：`functions` 参数传 JSON schema，模型返回 `function_call: {name, arguments}`，单轮单 function。
- **OpenAI Tool Use（2023 底更名）**：`tools` 参数，支持**并行调用**（`tool_calls: [...]`），消息类型新增 `tool` role。
- **Anthropic Tool Use**：语义相同，但 API 字段名不同（`tool_use` block），且 strict mode 约束更强。
- **MCP（Model Context Protocol）**：Anthropic 2024 推出的**标准化工具协议**，工具接口脱离 LLM 厂商绑定，我们项目用的是这个。

**项目里的选择：** MCP 的原因是 (1) 工具和 Agent 解耦，可独立部署，(2) 协议标准化易于换 LLM 厂商，(3) 原生支持 timeout / error envelope。

**陷阱提示** ⚠️：别说"MCP 是协议"就完，面试官可能问 "MCP 的 transport 是什么"，答：**Streamable HTTP / stdio**，我们项目用 HTTP。

### Q69. Multi-Agent 和 Single Agent with Tools 的取舍？什么时候该用哪个？

**关键词**: multi-agent, single agent, orchestration tradeoff

**骨架回答：**
- **Single Agent with Tools**：一个 LLM 轮询调工具。优点：**简单、便宜、debug 容易**。缺点：长上下文干扰、专业化不够。
- **Multi-Agent**：多个专业化 LLM 协作。优点：**职责清晰、并行化、专家 prompt**。缺点：**token 成本 N 倍、调试噩梦、通信损耗**（state 在 Agent 间丢信息）。

**什么时候用 Multi-Agent：**
1. 任务**天然可分解**且子任务差异大（我们项目：检索 vs 写作 vs 审核）
2. 子任务可**并行**（DeepScout 3 子 Agent 并行省时间）
3. 每个子任务**专家化 prompt** 能显著提升质量（CriticMaster 的 prompt 专注挑错）

**项目里为什么不是 Single Agent：** 核心原因是**并行化**——Single Agent 顺序跑 RAG+Web+SQL 是 ~30s，3 个 DeepScout 子 Agent asyncio.gather 是 ~10s，省 20s。其次是 CriticMaster 的独立 prompt 能发现 LeadWriter 自检发现不了的问题（类似 code review）。

### Q70. Token 使用优化的手段？你项目里怎么控成本？

**关键词**: token optimization, context compression, caching

**骨架回答：**
手段：
1. **Prompt 压缩**（LLMLingua 能压 10x）
2. **摘要式记忆**（累计对话做 running summary）
3. **工具描述精简**（MCP tool schema 不要写冗长 description）
4. **KV Cache**（OpenAI 的 prompt caching，Anthropic 的 cache_control）
5. **结构化输出** 而非自由文本（少废话）
6. **模型分级**（便宜模型做分类，贵模型做推理）

**项目用了：**
- **多 Key 路由 llm_router.py**：实际是**负载分担** + 并发加速，不是成本优化，但相当于让每个 Agent 用独立实例不抢限流
- **CriticMaster 输出结构化 JSON**（score / issues / verdict），不生成散文
- **report_cache (md5(question) → answer)**：命中直接返回省全部 LLM token
- **demo_mode 跳过 Critic 和 RE_RESEARCHING**：节省 30%+ token

**没做的：** Prompt Caching——本项目 MiniMax-M2.5 不支持原生 prompt cache，如果换 Claude 可以上 `cache_control: ephemeral` 省大量 system prompt token。

### Q71. Temperature 参数对不同任务的影响？项目里各 Agent 怎么设？

**关键词**: temperature, sampling, determinism

**骨架回答：**
- T=0：**贪心解码**，最确定，适合分类、抽取、SQL 生成
- T=0.3-0.5：**轻度随机**，适合 summarization、改写
- T=0.7-1.0：**创意任务**，适合头脑风暴、多样化生成
- T>1.0：基本只在 N-best 采样做多样性时用

**项目设置：**
- **Router（分类）**：T=0 确定性优先
- **Text2SQL 生成**：T=0——SQL 必须精确
- **DeepScout 查询展开**：T=0.5 需要一点多样性但不能跑偏
- **LeadWriter / Synthesizer（写作）**：T=0.3 报告需要可读性但求事实准确
- **CriticMaster（审核）**：T=0 审核要确定性输出

**陷阱提示** ⚠️：**不要说"我全用 0.7"**——这是外行答案，面试官会追问"那 Router 也 0.7？你不担心分类不稳定？"。

### Q72. Agent 评测指标有哪些？你项目里怎么评测？

**关键词**: groundedness, faithfulness, end-to-end, step-wise, ragas

**骨架回答：**
Agent 评测维度：
1. **End-to-end 正确性**：最终答案对不对（我们 28/30 就是这个）
2. **Groundedness / Faithfulness**：答案是否有来源支撑（RAGAS 指标）
3. **Step-wise 正确性**：每步中间状态对不对（Reflector confidence）
4. **延迟**（P50 / P99）
5. **Token 成本**
6. **工具调用成功率**（MCP call success rate）
7. **幻觉率**（人工抽检）

**项目用了：**
- End-to-end：`tests/test_resume_metrics.py` 4 大类 30 小题
- 延迟：demo 49.6s / 真实 ~50-80s
- MCP success rate：MCPCallError 统计（日志级）
- Groundedness：没上 RAGAS，做简化——Synthesizer 强制引用 [Source N]，人工抽检引用一致性

**追问预演：**
- Q: 怎么没上 RAGAS？→ A: RAGAS 需要额外 judge LLM 批量评分，成本高，MVP 阶段用人工抽检替代，后续生产要补。

### Q73. Agent 系统的幂等性和可重试性怎么做？

**关键词**: idempotency, retry, side effect

**骨架回答：**
幂等性关键：**同样输入多次调用结果一致**，尤其对"会产生副作用"的操作（写数据库、发邮件）。

Agent 场景下的做法：
1. **工具层**：SQL 查询天然幂等（SELECT）；archival_memory_insert 要做 dedup（我们用 uuid 避免）
2. **网络调用**：设置 idempotency key 防止重试重复下单（本项目不涉及交易）
3. **重试策略**：指数退避（LLM 429 限流），最大 N 次
4. **写操作幂等化**：核心记忆 replace 而非 append（replace 天然幂等）

**项目里：**
- LLM 调用：llm_router.py 有 fallback key 重试
- MCP 调用：mcp_client.py 的 MCPCallError 带 retryable flag
- 记忆写入：Core Memory replace 幂等、Archival Memory 每次新 uuid 不冲突但**多次插入会有重复**——这是当前缺陷，没做 content-hash dedup

**陷阱提示** ⚠️：主动承认 Archival dedup 缺失，表现对系统缺陷的认知。

### Q74. Milvus / Qdrant / Weaviate / Chroma / Pinecone 对比？

**关键词**: milvus, qdrant, weaviate, chroma, pinecone, vector db

**骨架回答：**
- **Milvus**（本项目选型）：C++ 实现、分布式、**亿级向量**、多种索引（IVF/HNSW/DISKANN）、etcd+MinIO 架构——**工业首选**。缺点：部署重。
- **Qdrant**：Rust 实现、单机性能强、HNSW 默认、**payload filtering 优秀**——适合中等规模。
- **Weaviate**：GraphQL 查询、模块化（embedding 内置）、适合快速原型。
- **Chroma**：Python 原生、内存级轻量——**本地开发/小项目**。
- **Pinecone**：SaaS、零运维——但**贵**且依赖外部服务。

**为什么选 Milvus：** (1) 能源行业文档规模增长预期大（从 MVP 100 万到生产 1 亿）Milvus 可扩；(2) 本地部署数据不出机房（客户合规要求）；(3) IVF_FLAT / HNSW 都支持方便调优。

**追问预演：**
- Q: 为什么不是 Chroma？→ A: Chroma 单机 SQLite 存储，百万级没问题但亿级要崩；且 Chroma 不支持标量过滤 + 向量混合查询。

### Q75. BGE-m3 / E5 / OpenAI ada-002 / Cohere 等 embedding 模型怎么选？

**关键词**: bge-m3, e5, ada-002, cohere, embedding

**骨架回答：**
- **BGE-m3**（本项目）：BAAI 开源、1024 维、**多语言（中英）**、MTEB 中文 SOTA、本地部署免费——**中文场景首选**。

**为什么选 BGE-m3：**

2. **本地部署**，客户数据不出机房（合规硬要求）
3. 1024 维 vs OpenAI 3072 维——**省 3x 存储和检索时间**
4. 免费

**追问预演：**
- Q: BGE-m3 比 bge-large-zh 强在哪？→ A: m3 是 **multi-function**（dense + sparse + colbert 三种检索模式一次性出来），同模型支持混合检索，减少 pipeline 复杂度。

### Q78. 项目最难的技术挑战是什么？

**关键词**: hardest challenge

**骨架回答：**
**Multi-Agent 的状态一致性。** 问题是 7 个 Agent 顺序 + 条件迭代跑，AgentState 里 critic_issues 被 Critic 写、被 Synthesizer 读，中间 DeepScout 并行 3 个子 Agent，各自改不同字段——调试时**报告里出现旧数据和新数据混合**，查了半天是因为我在 DeepScout 里直接 mutate state 而不是返回 diff 让 LangGraph reducer 合并。改成 `return {"scout_results": [...]}` 让框架合并后问题消失。教训：**LangGraph 状态更新必须用返回值，不能 inplace mutate**，因为并行分支会覆盖。

**陷阱提示** ⚠️：不要答"环境配置难"——这是**初级信号**。挑一个**设计层面**的困难。

---

### Q79. 最自豪的设计决策是什么？

**关键词**: proudest design decision

**骨架回答：**
**MCP 三层降级架构。** 传统 Agent 遇到工具故障要么全盘崩溃要么静默失败，我做了三层隔离：(1) **Agent 层** try/except 单工具失败不影响其他；(2) **MCPClient 层** 标准化 error envelope 区分 retryable / fatal；(3) **Pipeline 层** `api_server.py:517` `try run_deep_research() except run_legacy_graph()` ——Graph 2 整体崩就退回 Graph 1 ReAct。这设计让系统在部分组件故障时**仍能给用户一个答案**而不是 500。工程价值大于技术难度。

---

### Q80. 项目里最大的失败或返工是什么？（Gap 说辞整合）

**关键词**: biggest failure, rework, honest

**骨架回答（整合 §2 Gap 说辞）：**
最典型的一次是**评测从 19/30 掉到 26/30 再掉到 25/30 再到 28/30**——Round 1 修了 verdict_kw bug 和关键词放宽，Round 2 加了非确定性处理反而让一个原来过的用例不过了（因为我宽松了判定反而引入了误报），Round 3 才修平。教训：**评测框架改动要有回归保护**，我当时没写每次修改前后的 diff 对比，改了后面引入新问题才发现。

另一层更深的反思：Round 1 定位到 **Router general 意图漏 RAG** 是根因，但改 Router prompt 会打翻 10+ 个原本过的测试，最后选择在评测侧标 ACCEPTABLE_PARTIAL 而非硬改 Router——这是**工程取舍**：局部正确 vs 全局稳定，但写进简历时"针对性修复"表述偏乐观，实际是**评测层面承认 + Router 不动**的混合策略。

**陷阱提示** ⚠️：**这是 Q22/Q23 的行为题版本**——一定不要说"我改了 Router prompt"，说"做了取舍标 ACCEPTABLE_PARTIAL"。

### Q81. 如果重新设计，会改哪里？

**关键词**: redesign, hindsight

**骨架回答：**
三处：(1) **评测前置**——项目初期就该写死 30 条 ground truth 测试集，先测后改，而不是到中后期加评测；(2) **MemGPT 真·自主化**——当前规则触发是妥协，换 Claude 后应改为真·function calling 让 LLM 自主决定；(3) **Archival Memory content-hash dedup**——当前多次写入相同内容会存多份，应该插入前查 hash 去重；(4) **Observability**——应该接 LangSmith 或自建 trace 记录每次 Graph 执行轨迹，debug 时不用瞪日志。

### Q82. 怎么保证代码质量？

**关键词**: code quality, testing, review

**骨架回答：**
(1) **测试分层**：单元（RAG 召回 / SQL 生成）+ 集成（test_resume_metrics.py 4 大类 30 题）+ 记忆回归（test_section2_memory.py MEM-1 to MEM-6）；(2) **Checkpoint 制度**——每轮测试完写 `docs/checkpoints/` 记录结果和迭代过程，Round 1 → Round 3 都有留档；(3) **Troubleshooting log**——每个 bug 记录 `docs/troubleshooting-log/issue-YYYYMMDD-NNN.md`，失败路径也留档防止重蹈覆辙；(4) **git commit 粒度细**——每个修复独立 commit，方便 bisect。

**没做的：** 没上 CI（GitHub Actions），没做 code coverage 统计——单人项目 MVP 阶段妥协，生产要补。

## §6 应急兜底模板

> 手册不可能覆盖所有题。以下是**卡壳时的 5 种标准应对**。

---

### 应急 1：买时间话术

> "这是个好问题。我结合我实际做过的部分讲——……"

**适用：** 题目偏理论，需要几秒组织语言。**原理：** 把题目锚定到**自己熟悉的项目部分**，边讲边想下一步。

---

### 应急 2：框架化答题（场景 → 约束 → 方案 → 权衡 → 结果）

> "这个问题我从几个层面看：(1) **场景是什么**——比如是单机还是分布式、QPS 多少、数据量多大；(2) **约束是什么**——比如延迟 / 成本 / 合规；(3) **方案选项**——A/B/C 各自的特点；(4) **权衡依据**——基于约束选哪个；(5) **结果**——实际跑下来怎么样。"

**适用：** 设计题（"你会怎么设计一个 X"）。**原理：** 展示**结构化思维**，即使细节不对，方法论不失分。

---

### 应急 3：降级回答（诚实承认 + 表达理解）

> "这部分我没深入做过，但我的理解是……。如果真遇到这个问题，我会先查 X / Y 文档，然后做一个小实验验证。"

**适用：** 完全不会的问题。**原理：** **坦诚不是减分项**——装懂被追问穿帮才是灾难。降级时**展示学习路径**（查什么资料、做什么实验）反而加分。

---

### 应急 4：反向提问（题意模糊时）

> "这个问题我想更精准回答，您是想了解 X 场景下的做法，还是 Y 场景？" / "您这里说的 [术语] 指的是 [解释 A] 还是 [解释 B]？"

**适用：** 题目模糊或有歧义。**原理：** 避免答错方向浪费机会。**但别滥用**——一次面试最多 1-2 次反问，多了显得慢。

---

### 应急 5：类比迁移（不熟的领域用熟悉的项目类比）

> "这个我没直接做过，但我项目里有一个类似的问题：……。类比过来，这里应该是……。"

**适用：** 面试官问了不在项目里的话题（比如 "你做过分布式事务吗"）。**原理：** **把战场拉回自己熟悉的地方**。即使类比不完全成立，面试官会看到你的**迁移能力**。

---

### 终极兜底：记住这三句话

1. "**我不确定，但让我推理一下**——" + 结构化分析
2. "**在我的项目里，类似的场景是**——" + 具体细节
3. "**这是个好问题，落地时需要权衡**——" + 列 trade-off
